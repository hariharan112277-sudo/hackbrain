# Unified Error Model Specification (error_model.md)

Every error response generated by the IOB system strictly enforces compliance with the **RFC 7807 Problem Details** standard specification.

## 1. Unified Structure Validation Schema

```python
class ValidationErrorDetails(ContractBaseModel):
    field: str = Field(..., description="Dot-notation pointer to the invalid field payload target location")
    message: str = Field(..., description="Human-readable constraint evaluation failure message context")
    rejected_value: str | None = Field(default=None, description="String representation of the rejected data fragment")

class ProblemDetailsResponse(ContractBaseModel):
    type: str = Field(..., description="URI references resolving to clear human-readable documentation concerning error states")
    title: str = Field(..., description="Static generic structural high-level operational text summary")
    status: int = Field(..., description="The standard baseline HTTP status code returned for this context")
    detail: str = Field(..., description="Granular application text detailing the direct operational exception root cause")
    instance: str | None = Field(default=None, description="Unique relative tracking URI reference path pointing down to the invocation target context")
    error_code: str = Field(..., description="Internal alphanumeric system failure state tracking classification token")
    correlation_id: str = Field(..., description="Global tracing cross-system infrastructure identifier tracking key")
    retryable: bool = False
    invalid_params: list[ValidationErrorDetails | dict[str, Any]] = Field(default_factory=list)
```

In code, `ProblemDetailsResponse` is an alias for `ProblemDetails` to keep compatibility with the previously generated Phase 2 model name.

## 2. Standardized Enterprise System Failure Mapping Matrix

| Operational Error Context | HTTP Code | Error Code | Detail Message Template |
| :--- | :---: | :--- | :--- |
| Payload Field Constraint Breach | 422 | `ERR_SYS_VAL_01` | "Provided field formatting constraints failed validation criteria checks." |
| Token Missing or Malformed | 401 | `ERR_SEC_ATH_01` | "Bearer authorization header missing, expired, or cryptographically invalid." |
| RBAC Authorization Violation | 403 | `ERR_SEC_AUT_02` | "User identity lacks administrative privileges for target resource node operations." |
| Resource Target Missing | 404 | `ERR_REP_NF_01` | "Requested asset or entity with tracking token identification could not be located." |
| Storage Transaction Timeout | 503 | `ERR_INF_DB_01` | "High-frequency telemetry persistent store failed to respond within strict execution window limits." |
| Internal Unhandled Fallback | 500 | `ERR_SYS_UNK_99` | "An unhandled execution state failure occurred within backend core layers." |

## 3. Programmatic Failure Mapping Contract

```python
class EnterpriseErrorCode(str, Enum):
    ERR_SYS_VAL_01 = "ERR_SYS_VAL_01"
    ERR_SEC_ATH_01 = "ERR_SEC_ATH_01"
    ERR_SEC_AUT_02 = "ERR_SEC_AUT_02"
    ERR_REP_NF_01 = "ERR_REP_NF_01"
    ERR_INF_DB_01 = "ERR_INF_DB_01"
    ERR_SYS_UNK_99 = "ERR_SYS_UNK_99"

class EnterpriseFailureMapping(ContractBaseModel):
    operational_error_context: str
    http_code: int
    error_code: EnterpriseErrorCode
    detail_message_template: str
```

The executable registry is exposed as `ENTERPRISE_FAILURE_MAPPING` in `integration.backend_contracts`.

## 4. Canonical Error Handling Rules

- All validation failures must use HTTP `422` with `ERR_SYS_VAL_01`.
- Authentication token absence/malformed states must use HTTP `401` with `ERR_SEC_ATH_01`.
- Authenticated but unauthorized RBAC violations must use HTTP `403` with `ERR_SEC_AUT_02`.
- Repository not-found outcomes must use HTTP `404` with `ERR_REP_NF_01`.
- Storage timeout/dependency failure states must use HTTP `503` with `ERR_INF_DB_01`.
- Unclassified failures must use HTTP `500` with `ERR_SYS_UNK_99` and must include a correlation id.
